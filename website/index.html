WarGame 2

Kinda-Sorta Basic Benchmarking in Java

Download the WarGame 2 source and class files.

To run the WarGame:

java App
This will run like so:
  1. Priming phase: 60 seconds of games running in attempt to mitigate overhead of setup and to reach a stable speed
  2. Testing phase: Determine if the speed is within a +/-00.01% range, check in (1/speed) seconds
  3. Retesting phase: if the speed is not within the range, test again in (1/speed) seconds
  4. Results phase: display the results

The WarGame program has three primary settings:

  • Threads: by default, the number of logical cores available
  • Confidence: the change allowed during the testing phase, as a percentage. By default, 00.01%.

    For example, if the speed at the beginning of the testing phase is .2345, then the confidence range would be
    0.23447655 < .2345 < 0.23452345

  • Prime time: by default, 60 seconds

    Prime time allows the program to reach a period of stability before testing begins.

Modes

The WarGame has various modes: Help, Short and Interactive.

Short

java App s

This mode will run the WarGame program with 1% confidence and a 10 second priming phase.

Interactive

java App i

This mode will run the WarGame program with user specified settings, those being threads, confidence and prime time.

Backstory

When I was in tenth grade, I spent a week or so coding the original WarGame — a simulator for playing war. The sim was fine, but my favorite past time was running it on various computers with a million games to see how long it would end up taking. In 2012, I built a new computer with an incredibly fast processor and I needed something better to test it than my original WarGame.

Thus, the WarGame 2 was made. The second generation WarGame is the same at its core -- the game of War simulating two people playing cards. But the advancements are in the details: multiple threads play simultaneously (or whatever is an approximate of that) and a testing method that scales based on hardware capabilities. Threading makes full use cores available while the scaling method used allows for some degree of precision and consistency between runs. A faster computer will run more games for longer to ensure that its speed isn't fooling anyone.

A note on the scaling method; after a minute, we time a test for 1/speed at that instant. On a slow computer, this will be only be another second or so of extra time. On a fast computer (like mine, or maybe a server), this ends up being thirty extra seconds of time, per test. Because the program use a 00.01% confidence range, there are often many tests. If the speed at the instant is not within the +/-00.01% range, the test starts over again with another (1/speed) seconds of time. Total times around ten minutes are not unheard of.
